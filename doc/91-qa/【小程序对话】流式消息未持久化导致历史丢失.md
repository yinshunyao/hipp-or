# 问题描述

小程序对话页使用流式发送后，当用户退出会话回到智能体列表，再次进入会话时看不到刚刚的历史消息（用户消息与助手回复均缺失）。

复现：进入 `pages/chat/chat` 发起一次 `POST /mp/chat/sessions/{id}/messages/stream`，等待回答结束后退出页面并重新进入同一会话，历史列表不包含本次对话。

# 根因分析

`/mp/chat/sessions/{id}/messages/stream` 由 `StreamingResponse` 返回生成器。  
原实现在生成器内直接使用请求依赖注入的 `auth.db` 写入用户消息与助手消息。该依赖会话受请求生命周期管理，流式场景下存在事务边界与响应体执行时序不一致，导致写入无法稳定提交，出现“发送成功但历史缺失”。

# 涉及文件

- `hipp-or/hipp-api/apps/mp_chat/crud.py`：流式链路改为独立 `session_factory` 事务，分别持久化用户消息与流结束后的助手消息/会话摘要。
- `hipp-or/hipp-api/apps/mp_chat/views.py`：调整流式调用参数，不再向生成器传递请求态 `db`。
- `hipp-or/doc/02-dr/04 小程序对话/小程序用户对话设计.md`：补充流式持久化事务边界说明。
- `hipp-or/doc/03-tr/04 小程序对话/小程序用户对话测试设计.md`：新增“退出重进历史可见”的回归用例。
- `hipp-or/doc/01-or/【小程序对话】用户智能对话——后端处理.md`：补充已完成问题项。

# 设计约束更新

流式对话必须采用“独立持久化事务”策略：
1. 流开始前先落库用户消息（独立事务提交）；
2. 流正常结束后再落库助手消息与会话摘要（独立事务提交）；
3. 不允许依赖 `StreamingResponse` 请求级会话的隐式提交保证历史一致性。
