# 问题描述

小程序用户执行「退出登录」后，从首页点击「需求咨询」或「商业咨询」，界面再次表现为已登录正式用户（头像、昵称等与退出前一致），与「已退出」预期不符。

复现：使用已绑定微信/手机号的账号登录 →「我的」退出登录 → 回到首页 → 点击「需求咨询」或「商业咨询」→ 观察「我的」或接口行为，登录态恢复。

# 根因分析

1. **后端**：`UserDal.get_or_create_mp_guest` 对已有 `wx_server_openid` 的用户一律复用记录，且注释写明「若该 openid 已绑定正式/微信用户则直接返回该用户」。因此退出后前端清除 token 再调用 `wx.login` + `POST /auth/mp/guest` 时，仍为**同一正式用户**签发 JWT。
2. **前端**：路由守卫在无 token 时会 `EnsureGuestToken` → 成功后对 `!isUser` 调用 `GetInfo`，用新 JWT 拉取用户信息并把 `isUser` 置为 true，与正式登录后表现一致。
3. **附带**：`common/utils/storage.js` 的 `clean()` 仅 `removeStorageSync` 未重置模块内 `storageData` 缓存，存在退出后仍可能读到旧缓存的理论风险；本次一并重置为空对象。

# 涉及文件

- `hipp-or/hipp-api/apps/vadmin/auth/crud.py`：`get_or_create_mp_guest` 仅允许 `user_type=mp_guest` 复用；非游客已绑定用户返回 `ValueError`，由 `mp_guest_login` 转为 `ErrorResponse`。
- `hipp-or/hipp-uni/common/request/api/login.js`：`mpGuestLogin` 请求增加 `custom.skipErrorToast`，避免预期内的「需先登录」在冷启动/换票失败时反复弹全局 Toast。
- `hipp-or/hipp-uni/common/request/request.js`：响应拦截器识别 `skipErrorToast` 时跳过 `toast`。
- `hipp-or/hipp-uni/common/utils/storage.js`：`clean()` 同步清空内存中的 `storageData`。
- `hipp-or/doc/02-dr/04 小程序对话/企业宣传首页与访客体验设计.md`、`hipp-or/doc/03-tr/...`：设计/回归用例同步。

# 设计约束更新

- `/auth/mp/guest` 仅服务于**纯游客账号**（`mp_guest`）；已绑定正式身份的用户必须通过 `/auth/login`、`/auth/wx/login` 等正式登录入口换票。
