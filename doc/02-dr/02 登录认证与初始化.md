# 功能设计

## 小程序登录入口

- 登录页**默认**为「手机号 + 短信验证码」：`获取验证码`、验证码输入、登录提交。
- 提供切换到「账号密码登录」的入口（与默认态二选一展示，具体交互由前端实现）。
- **不再提供**「微信授权登录」入口；`hipp-uni` 登录页移除微信登录按钮及相关弹层。
- 账号密码分支：输入框字段名沿用 `telephone`（兼容历史客户端），语义为「手机号或账号」；密码默认掩码，支持显隐切换（仅前端展示态切换，不改变值）。

## 账号密码登录（`POST /auth/login`，`method=0`）

- 请求保持现有结构：`telephone` + `password` + `method=0` + `platform=1`。
- 登录标识解析规则保持现有实现：
  - 合法手机号：按 `telephone` 精确匹配；
  - 含字母账号：先 `name` 精确匹配，再 `nickname` 精确匹配。
- 失败文案继续使用统一口径：
  - 账号不存在/密码错误：`账号或密码错误`；
  - 冻结：`此账号已被冻结`；
  - 无权限：`此账号无权限`。

## 短信验证码登录（`POST /auth/login`，`method=1`，`platform=1`）

- 请求字段沿用现有约定：`telephone` 为手机号；`password` 字段承载**短信验证码**（与历史接口一致）。
- 发送验证码：登录场景（小程序与管理后台短信登录）统一调用 **`POST /auth/sms/send`**，请求体 JSON：`{ "telephone": "<11位手机号>" }`；不要求手机号已注册；底层仍用 `utils/sms/code.CodeSMS`（Redis 存码、频控等）。**禁止**将登录发码指向 `POST /vadmin/system/sms/send`（该路由为历史实现：`telephone` 走 query，且要求库中已存在该手机号，易与小程序自助注册发码混淆）。
- **处理顺序调整（相对原「先查用户再验码」）**：对 `platform=1` 且 `method=1`，应先校验短信验证码正确且未过期；校验通过后再按手机号查询用户。
  - **已存在用户**：继续走与现有一致的登录校验（拉黑、冻结、`is_staff` 等），通过后颁发 token。
  - **不存在用户**：**自动创建用户**后颁发 token（仅当验证码校验已通过）。新建用户默认建议：
    - `user_type`：使用 `system`（短信自助注册与系统用户同源管理），或与现有枚举一致的独立取值须在实现与迁移说明中统一，**不得**再依赖微信登录写 `wechat`；
    - `telephone`：本次手机号；
    - `name` / `nickname`：按手机号派生默认展示名（如「用户」+ 后 4 位，规则与现网数据风格一致即可）；
    - `is_staff=True`、`is_active=True`；`password`：随机哈希，兼容 token 载荷中的 `password` 校验；
    - `wx_nickname` / `wx_avatar` 可空；不要求绑定 `wx_server_openid`。
- `platform=0`（管理端等）的 `method=1` 行为：若仍需「仅已存在用户可短信登录」，可在服务端按 `platform` 区分：**仅 `platform=1` 启用自动建号**，避免后台账号被误创建（实现时以代码分支为准）。

## 微信授权登录（`/auth/wx/login`）与小程序

- **小程序前端**：不再调用 `/auth/wx/login`。
- **服务端**：接口可暂时保留供旧版客户端或运维排障，但新需求不以微信登录为交付路径；若后续完全下线，另立变更单删除路由与相关校验。

## 用户类型字段（`vadmin_auth_user`）

- 现有字段 `user_type`、`wx_nickname`、`wx_avatar` 继续用于历史数据与展示。
- 通过**短信自动注册**的新用户：`user_type` 建议为 `system`（与「后台或密码创建的用户」同一管理口径）；若历史已有大量 `wechat` 用户，并存期间列表与标签逻辑不变。

## 登录响应扩展

- `/auth/login` 返回体继续包含 `user_type` 等字段，供前端识别。
- 兼容保留：`access_token`、`refresh_token`、`token_type`、`is_reset_password`、`is_wx_server_openid`。

# 数据模型

- 表：`vadmin_auth_user`
- 新增字段：
  - `user_type` `String(16)` 非空，默认 `system`；
  - `wx_nickname` `String(128)` 可空；
  - `wx_avatar` `String(500)` 可空。
- 迁移方式：通过 Alembic 增量迁移新增字段；历史用户回填 `user_type=system`。

# 约束与边界

- 短信登录依赖阿里云短信（或当前项目已接入的短信通道）与 Redis 验证码存储；通道未配置时须返回明确错误，不得静默失败。
- 自动建号仅发生在 **验证码已验证通过** 之后，避免未验证手机号占用库表。
- 现阶段不引入数据库外键或新关联表；用户类型信息直接存储在用户表字段中。
- 用户类型仅用于来源区分与业务判定，不影响现有角色权限模型。

## 启动初始化补充（`hipp-api/run.sh`）

- 启动前继续通过核心表存在性检查决定是否执行 `init`。
- 当目标数据库连接返回 MySQL `1049 (Unknown database)` 时，脚本应自动执行建库：
  - 使用相同主机、端口、用户名、密码建立不指定库名的连接；
  - 执行 `CREATE DATABASE IF NOT EXISTS <db_name> CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci`。
- 建库成功后重连目标库并恢复原流程：
  - 核心表存在：直接启动服务；
  - 核心表缺失：执行迁移和种子初始化后启动服务。
- 仅数据库不存在场景启用自动建库；权限不足、网络不可达、认证失败等异常仍保持失败退出。

# 前端（hipp-uni）

- 登录页默认区：手机号、验证码、`获取验证码`（倒计时防刷）、登录按钮；错误与发送结果 Toast/文案与后端一致。
- 提供切换到账号密码登录；密码框显隐切换行为不变。
- 移除微信登录按钮、`getPhoneNumber` 登录链路、与微信一键登录相关的弹层与接口调用。
- 登录成功后仍持久化 `token`、`refreshToken`、`userType` 等，与现有请求拦截器一致。
