# 功能设计

## 范围与约定

- **鉴权**：小程序与现有移动端一致，使用 `Authorization: Bearer <access_token>`（`/auth/login` 等颁发的 JWT）。接口使用 `AllUserAuth`，仅登录用户可访问。
- **智能体来源**：仅允许使用 `vadmin_agent` 中 `status=published` 且未软删除的记录；服务端根据 `agent_id` 读取 `api_server`、`app_key`，**禁止**由客户端传入 Dify 地址或密钥。
- **历史会话可读、发送受限**：若会话关联智能体变为「未上架」或「已删除」，会话详情与历史消息仍可查询；但 `POST /messages` 与 `POST /messages/stream` 必须拒绝发送。
- **进行中的会话**：由小程序使用本地存储记录 `last_active_session_id`（可选）；进入「对话」Tab 时若存在且会话仍有效则 `navigateTo` 对话页，否则展示收件箱列表。
- **Dify 调用**：使用官方 Chat 应用接口 `POST {api_server}/v1/chat-messages`。小程序主路径为 **`response_mode=streaming`**，服务端按 SSE 行转发给客户端；对 `message/agent_message.answer` 的首行执行「结束状态 True/False（兼容 `:` / `：`）」过滤，避免状态标记进入实时可见正文。流结束后解析累积文本并落库 assistant 消息、更新 `dify_conversation_id` 与 `last_message_preview`。兼容保留 `response_mode=blocking` 的同步接口供非流式环境降级。请求体 `user` 使用 `user-{vadmin_auth_user.id}` 作为 Dify 侧用户标识；首次不传 `conversation_id`，后续使用流式或 blocking 响应中的 `conversation_id` 写入会话表以实现多轮连续。
- **流式持久化事务边界**：`POST /messages/stream` 的响应体执行晚于路由返回，消息落库不得依赖请求级依赖事务隐式提交。实现上需使用独立数据库会话事务：流开始前先提交 user 消息；流正常结束后再提交 assistant 消息与会话摘要，确保“退出再进入”历史可见。

## 后端（hipp-api）

### 模块与路由前缀

- 目录：`hipp-or/hipp-api/apps/mp_chat/`
- 路由前缀：`/mp/chat`（在 `application/urls.py` 注册）
- 标签：`小程序对话`

### 数据模型

**约定：库表不建数据库级外键（FOREIGN KEY）**

- `user_id`、`agent_id`、`session_id` 仅保存**逻辑关联**的整型 ID，与 `vadmin_auth_user`、`vadmin_agent`、`vadmin_chat_session` 的对应关系**不在 MySQL 层声明外键**。
- 参照完整性、级联删除等由**业务代码**保证；查询需要关联展示时在 Service/CRUD 层使用 **JOIN**（或 SQLAlchemy `join` / `joinedload` + 显式 `primaryjoin`，见 `apps/mp_chat/models`）。
- 优点：避免强约束与迁移/分库/批量清理时的锁与顺序问题；与「数据访问用 SQL/JOIN 自控」的约定一致。

**表 `vadmin_chat_session`（用户与智能体会话）**

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| id | PK | 主键 |
| user_id | Integer，**非 FK** | 对应 `vadmin_auth_user.id`（逻辑引用） |
| agent_id | Integer，**非 FK** | 对应 `vadmin_agent.id`（逻辑引用） |
| title | String(255) | 展示标题，默认智能体名称，可修改 |
| agent_name_snapshot | String(255) nullable | 智能体名称快照（创建会话时写入；源智能体删除时用于历史展示） |
| agent_avatar_snapshot | String(500) nullable | 智能体头像快照（可选） |
| last_message_preview | String(500) nullable | 最近一条消息摘要 |
| is_pinned | Boolean default false | 是否置顶 |
| dify_conversation_id | String(255) nullable | Dify 返回的 conversation_id |
| is_topic_closed | Boolean default false | 话题是否已结束（助手首行「结束状态 True」归档后设为 true；延续话题后为 false） |
| + BaseModel 字段 | | create/update/delete_datetime、is_delete |

建议保留普通索引：`user_id`、`agent_id`（便于按用户/智能体筛选；**非**外键索引）。

**表 `vadmin_chat_message`（会话消息缓存）**

| 字段 | 类型 | 说明 |
|:---|:---|:---|
| id | PK | 主键 |
| session_id | Integer，**非 FK** | 对应 `vadmin_chat_session.id`（逻辑引用） |
| role | String(20) | `user` / `assistant` |
| content | Text | 文本内容 |
| + BaseModel 字段 | | 同上（消息可软删除以便与会话一并隐藏） |

建议保留普通索引：`session_id`（便于按会话拉消息；**非**外键索引）。

说明：消息落库用于列表预览与历史展示；与 Dify 侧会话通过 `dify_conversation_id` 对齐。若历史库中曾创建外键，通过迁移脚本删除约束即可，表结构仍以「无 FK + 索引」为准。

### API 摘要

| 方法 | 路径 | 说明 |
|:---|:---|:---|
| GET | `/mp/chat/inbox` | 收件箱：`q` 可选；**`kind=session`**（小程序对话 Tab 使用）时**仅**返回已归档话题会话行（`kind=session`，`is_topic_closed=true`），**不**返回 `kind=agent`；**省略 `kind`** 时返回合并列表（已上架智能体入口行 + 会话行），便于调试或其它客户端 |
| GET | `/mp/chat/agents/{agent_id}/archived-topics` | 某智能体下已归档会话分页：`limit`（默认 20，最大 50）；可选游标 `before_update_ts` + `before_session_id`（须成对）表示当前列表**最旧一条**，用于拉取**更旧**一页。响应 `data.items[]`：`session_id`、`display_title`、`update_ts`；`has_more`。`items` 为时间正序（旧→新）。 |
| POST | `/mp/chat/sessions` | body: `{ agent_id }`，创建会话 |
| GET | `/mp/chat/sessions/{id}` | 会话详情（含智能体展示字段） |
| PATCH | `/mp/chat/sessions/{id}` | body: `{ title?, is_pinned?, resume_topic? }`；`resume_topic: true` 时将 `is_topic_closed` 置为 false，延续同一 Dify 线程 |
| DELETE | `/mp/chat/sessions/{id}` | 软删除会话 |
| GET | `/mp/chat/sessions/{id}/messages` | 分页消息列表 |
| POST | `/mp/chat/sessions/{id}/messages` | body: `{ query }`，转发 Dify（blocking）并写入双方消息 |
| POST | `/mp/chat/sessions/{id}/messages/stream` | body: `{ query }`，Dify **streaming** SSE 透传；流正常结束后落库 assistant |
| WS | `/mp/chat/sessions/{id}/messages/ws?token=...` | 小程序真机流式主链路：握手后客户端发送 `{query}`，服务端按增量推送并在结束时下发 topic 状态 |

会话相关返回补充约定：
- `GET /mp/chat/inbox` 的 `kind=session` 行包含 `session.agent_status`，取值 `active/offline/deleted`；当请求带 **`kind=session`** 时，`session.is_topic_closed` 恒为 `true`（该模式仅返回已归档话题）。不带 `kind` 的合并列表中，`kind=session` 行可能含未归档会话。
- `GET /mp/chat/sessions/{id}` 返回 `agent_status` 与 `agent` 展示对象；当智能体已删除时，`agent` 使用会话快照字段兜底；包含 `is_topic_closed`。
- **C 端展示标题 `display_title`（与库字段 `title` 并存）**：当用户未自定义会话标题（`title` 仍与 `agent_name_snapshot` 一致或等价于创建时的智能体名场景）时，`display_title` 取**当前**智能体 `name`，以便管理端修改智能体名称后小程序收件箱与对话页导航栏立即一致；若用户已通过 `PATCH` 修改过标题（`title` 与 `agent_name_snapshot` 不一致），`display_title` 与 `title` 相同，保留用户命名。小程序优先用 `display_title` 展示列表名与导航栏标题。
- **C 端展示头像 `display_avatar_url`（新增）**：`GET /mp/chat/inbox` 的会话行需提供对话对方头像 URL。优先级：人工客服会话取客服头像 > 智能体当前头像 > 会话快照头像（`agent_avatar_snapshot`）> 默认占位头像。前端统一使用该字段渲染列表左侧头像，禁止退化为首字母块。
- `POST /mp/chat/sessions` 返回结构与会话详情保持一致（便于前端进入会话后直接使用状态字段）。
- `POST /mp/chat/sessions/{id}/messages`（blocking）成功时返回体包含 `topic_closed: bool`（与流式语义一致）。

### 收件箱规则（对话 Tab：`kind=session`）

1. **范围（对应 OR `【小程序对话】用户智能对话.md` + `【小程序对话】需求与商业场景页.md`）**：`GET /mp/chat/inbox?kind=session` 的 `items` **全部为** `kind=session`，且对应会话满足 `is_topic_closed=true`、`is_delete=false`。**不得**返回 `kind=agent` 行。进行中会话、尚未开聊的智能体入口均在「需求」「商业」等场景页处理，不在带 `kind=session` 的请求中出现。
2. **无 `kind` 或省略**：返回合并收件箱（已上架智能体 `kind=agent` + 其余会话 `kind=session`），与历史默认行为一致。
3. **`kind=session` 查询**：当前用户未删除且 `is_topic_closed=true` 的会话，左连接 `vadmin_agent`（用于 `agent` 片段、`agent_status`、头像与 `display_title` 计算）。
4. **排序（仅 `kind=session`）**：先按是否置顶分区——**置顶**（`is_pinned=true`）的会话整体在前、**非置顶**在后；各分区内按 `update_datetime` 倒序（新近活跃在前），同时间以 `id` 大者优先（与 `inbox_order.sort_session_pairs_by_recency_desc` 一致）。
5. **搜索 `q`（`kind=session`）**：对会话 `title`、`last_message_preview`、`agent_name_snapshot`、关联智能体实时 `name`（若存在）做包含匹配；仅过滤上述归档会话集合。
6. **小程序 `pages/chat/index`**：`getChatInbox` 固定传 `kind=session`；列表仅渲染会话行；点击行进入 `subpkg/chat/chat`；空列表时提示「暂无归档话题」类文案（产品可微调）。

### 双智能体入口与菜单语义

- 小程序 C 端固定两类**进行中对话**入口：`需求`（超级架构师）与 `商业`（连续创业者）；「对话」Tab 仅承载**归档话题**列表，不展示智能体固定行。
- 前端进入任一入口时，必须先调用后端“场景解析智能体”接口，由后端按入口类型过滤/定位目标智能体；不允许客户端仅依赖智能体名称（如若干 candidates）自行匹配或兜底。客户端也不允许上传 Dify 地址或密钥。
- **场景解析智能体接口（强制）**：
  - 方法：`GET /mp/chat/scene-agent?scene=requirement|business`
  - 语义：后端在当前“可服务”集合内（`vadmin_agent.status=published` 且未软删除，且已通过测试）按场景映射的 **`service_type`** 过滤；若同类型候选 > 1，则在同类型内 **随机选定 1 个**，并返回该智能体信息。
  - 场景与类型映射（与 OR 一致）：
    - `scene=requirement` → `service_type=需求分析`
    - `scene=business` → `service_type=商业评估`
  - 返回（建议）：
    - `service_type`：本次解析使用的类型
    - `agent`：智能体展示片段（至少含 `id/name/icon/icon_background/icon_url/status/service_type`）
    - `session`（可选）：若当前用户在该 `agent_id` 下存在进行中会话（`is_topic_closed=false` 且未软删除），可一并返回，便于场景页直接续聊；若无则为 `null`
  - 无可服务智能体：返回明确业务错误（例如 422 或业务码），前端展示“当前场景暂不可用”并禁用发送/创建会话。
  - 随机策略：随机仅在“同类型多候选”时生效；实现需避免因名称变更导致映射漂移。
- 场景页内已归档话题（`is_topic_closed=true`）在历史区以标题形式展示，点击后进入对应会话详情；与「对话」Tab 列表数据来源一致（均为归档会话），但场景页按智能体维度与时间线组织，对话 Tab 为跨智能体归档汇总。

### 话题归档

- **判定**：对助手完整回复（流式累积的 `full_answer`，blocking 的 `answer`）在去除首行「结束状态」**之前**，若首行匹配 `结束状态 True`，则本轮结束后将会话 `is_topic_closed` 置为 `true`。匹配规则须兼容：
  - `结束状态true` / `结束状态 true` / `结束状态: true` / `结束状态：True`
  - `True/False` 大小写不敏感（`true`、`TRUE`、`False` 均可识别）
  - `结束状态` 与布尔值间允许无空格或任意空白
- **落库正文**：仍经 `strip_finish_status_prefix` 后再写入 `vadmin_chat_message` 与 `last_message_preview`。
- **标题（展示名与 Dify `name` 对齐）**：
  - 每轮 **`POST /v1/chat-messages` 成功**（blocking 或流式结束落库）后，**优先**从 blocking **响应体**解析会话 `name`（见 `extract_conversation_name_from_chat_response`：顶层 `name` 或 `conversation.name` 等）。
  - 若无则凭 `conversation_id` 调用 **`GET /v1/conversations`**，在 `data[]` 中用 **`id` 与当前 `conversation_id` 匹配**（比对时忽略大小写与连字符差异），取 **`name`** 写入 `vadmin_chat_session.title`（最长 255）。
  - **话题归档**（`is_topic_closed` 置为 true）且仍无法取得 Dify `name` 时，才允许短回退：若 `title` 仍与 `agent_name_snapshot` 一致，用本轮用户 `query` 的**短摘要**（`_preview`，约 36 字）；若用户此前已改过标题则保留原 `title`。禁止用长段用户首问作为列表展示标题。
- **SSE 元事件**：流式路径在 Dify 增量结束后、**会话与 assistant 落库成功之后**，追加一行 SSE：`data: {"event":"mp_topic","topic_closed":true|false}\n\n`，供小程序在**全部正文展示完成后**再提示「话题已结束」等（`topic_closed` 与库内 `is_topic_closed` 一致）。
- **发送拦截**：`is_topic_closed=true` 且智能体仍为 `active` 时，`POST .../messages` 与 `.../messages/stream` 在调用 Dify 前拒绝，业务码 **`46002`**（与智能体不可用 `46001` 区分）。
- **延续话题**：`PATCH .../sessions/{id}` 带 `resume_topic: true` 将 `is_topic_closed` 置为 `false`，不清空 `dify_conversation_id`，用户可在同一话题下继续多轮。

### 发送可用性校验

1. 对 `POST /mp/chat/sessions/{id}/messages` 与 `POST /mp/chat/sessions/{id}/messages/stream`，在写入用户消息前执行会话可发送校验。
2. 校验规则：
   - `vadmin_agent` 存在且 `status=published` 且未软删除 -> 可发送；
   - `vadmin_agent` 存在但非 published 或已软删除 -> 拒绝发送并返回业务错误；
   - `vadmin_agent` 不存在（物理删除） -> 拒绝发送并返回业务错误。
3. 被拒绝时不调用 Dify；返回业务错误码 `46001`（智能体不可发送）与展示文案，供前端禁用输入框与提示。
4. 若会话 `is_topic_closed=true`（话题已归档）：拒绝发送，业务码 **`46002`**。前端应将其视为“当前会话不可继续发送”，并引导用户开启新话题（场景页/会话页可自动新建会话并发送），或在会话列表使用「继续对话」延续原话题（`resume_topic`）。

### Dify 错误处理

- HTTP 非 2xx 或响应无 `answer` 时返回业务错误信息，不落库 assistant 消息（可选记录 user 消息，实现时以一致性优先：失败时仍保存 user 消息并返回错误文案由前端展示）。

## 前端（hipp-uni）

### 启动与默认进入页

- 鉴权成功后默认进入 `pages/requirement/index`（见 `hipp-uni/permission.js` 的 `indexPage`），不再进入 `pages/index`。
- 路由守卫中登录态用户访问登录页时，与 `indexPage` 一致重定向，保持首次启动与登录后行为一致。

### 页面与路由

- `pages/chat/index`：对话 Tab **归档话题**收件箱（标题「对话」、搜索框、列表、长按会话菜单：置顶或取消置顶、重命名、删除；分享使用小程序分享能力，路径带 `sessionId`）。**置顶菜单项**：默认 `is_pinned=false` 时该项文案为「置顶」，点击后 `PATCH` 置 `is_pinned=true`；已置顶时该项为「取消置顶」，点击后置 `false`（同一槽位随状态切换，不得并列两条）。列表项均为已归档会话，展示「已结束」等弱标记；长按菜单含「继续对话」（`PATCH` `resume_topic` 后进入会话）。
- **收件箱列表项**：每行左侧头像、右侧为标题行 + 预览行。头像必须使用 `display_avatar_url`（表示对话对方头像）渲染；仅当 URL 加载失败时回退默认占位头像。标题为会话展示名（`display_title`），**最多两行**，超出用省略号截断（`-webkit-line-clamp: 2` 等价效果）；状态标签（已下架、置顶、已结束等）位于标题行**右侧**，与标题同列 flex 布局，标题区 `flex:1; min-width:0` 保证截断生效。
- `subpkg/chat/chat`：单会话聊天（`onLoad` 参数 `sessionId`），导航栏标题取接口 `display_title`（无则 `title`）；`onShow` 刷新会话元数据以便后台改智能体名称后返回该页可见更新；返回 `navigateBack` 到收件箱。
- 若会话 `agent_status != active`，页面进入只读模式：显示历史消息；输入框禁用、发送按钮禁用，并展示「智能体已下架/删除，无法继续对话」。

### 对话页底部输入区（`subpkg/chat/chat`）

- **组件**：使用 `textarea`（非单行 `input`），开启 `auto-height`，长文本自动换行；设置合理 `max-height` 与 `overflow-y: auto`，避免占满全屏。
- **发送键位**：`textarea` 的普通回车仅用于换行；不绑定回车直接发送。发送动作由发送按钮触发为主；在支持实体键盘的端（如 H5/开发工具）支持 `Ctrl/Cmd + Enter` 组合键发送。
- **键盘**：依赖微信小程序 `textarea` 的 `adjust-position`（默认开启），使输入焦点随软键盘上推；可配合 `cursor-spacing` 与底部留白，减少光标贴键盘。
- **视觉**：整体为浅色底栏 + 白底圆角容器（大圆角）、细边框与轻阴影；发送为**圆形主色按钮 + 向上箭头图标**（无「发送」文字），空内容或加载中禁用并置灰。
- **安全区**：输入栏 `padding-bottom` 叠加 `env(safe-area-inset-bottom)`，避免与 Home Indicator 重叠。

### 消息区展示、乐观更新与发送状态（`subpkg/chat/chat`）

- **消息区宽度**：用户消息气泡保持右对齐、**最大宽度约 80%**（与常见 IM 一致）。**助手消息**因正文通常较长，气泡在横向上**铺满屏幕可视区域**（抵消消息列表容器左右内边距，左右与屏幕边缘对齐），避免右侧大块留白；样式上与用户消息区分。
- **纯文本气泡（用户消息）**：用户消息正文使用 `<text>`（或等价可继承样式的文本节点）展示。**禁止**用 `rich-text` 的自定义 nodes 拼装「伪标签名」承载纯文本：微信小程序 `rich-text` 仅支持有限标签（如 `div`/`span`/`p` 等），使用非法 `name` 会导致正文不渲染、气泡塌缩，表现为「仅有色块/占位无文字」。
- **乐观展示**：用户点击发送后，**立即**向本地 `messages` 追加一条 `role=user` 与一条空的 `role=assistant`（流式填充），清空输入框；列表 `key` 使用服务端 `id`（历史消息）或客户端 `localId`（本次发送生成），避免仅用 `index` 导致重渲染异常。
- **发送状态（仅本次会话内乐观追加的用户消息）**：
  - `sending`：请求进行中；可在用户气泡下方展示「发送中…」。
  - `sent`：流式或 blocking 成功结束且无未捕获异常时写入。
  - `failed`：请求失败时写入；用户气泡下方展示「重试」，点击后**以同一条用户文案**重新调用 `POST .../messages/stream`（失败则回退 blocking），并清空对应助手气泡后重新填充。
- **历史消息**：从 `GET .../messages` 拉取的消息无 `sendStatus` 字段，不展示发送中/重试，仅展示气泡正文。
- **Markdown 展示策略（助手消息）**：
  - 用户消息保持纯文本 `<text>` 展示，避免样式与语义混淆。
  - 助手消息支持 Markdown（标题、列表、引用、行内/块代码、链接、**表格**）渲染，前端将 Markdown 转为小程序 `rich-text` 可渲染的 **HTML 片段**后展示（优先使用小程序原生渲染能力，不依赖 WebView）。
  - **表格**：表体每一数据行必须生成完整 `<tr><td>…</td></tr>`（含行首 `<tr>`）。省略 `<tr>` 时，微信 `rich-text` 易出现表体多行横排成多列、单元格极窄等错位。
  - 渲染前需进行基础安全处理：先转义原始 HTML，再按 Markdown 语法生成受控标签；链接仅允许 `http/https`，其余协议按纯文本处理，避免注入风险。
  - 当 Markdown 解析结果为空时，保持「（无内容）」兜底文案，确保流式和 blocking 分支行为一致。

补充：`pages/requirement/index` 与 `pages/business/index` 的助手消息渲染策略与 `subpkg/chat/chat` 保持一致（同一套 Markdown → HTML（`rich-text`）转换与安全约束），避免同一智能体回复在不同入口展示不一致。

### 请求封装

- 在 `common/request/api/mp/chat.js` 中封装上述 API，使用既有 `request` 实例与 Token 拦截器。
- **流式**：微信小程序使用 `uni.request` + `enableChunked: true` + `onChunkReceived` 接收分块，按行解析 Dify `data:` JSON，累积 `message` / `agent_message` 的 `answer` 增量更新气泡；不支持分块的环境回退为 `POST .../messages`（blocking）。
- **小程序真机流式改造（WSS）**：
  - 保留既有 `sendChatMessageStream` 入口与页面调用方式；
  - 在 `MP-WEIXIN` 分支优先使用 `uni.connectSocket` 连接 `wss://<host>/.../messages/ws?token=<jwt>`；
  - 连接建立后发送 `{"query":"..."}`，服务端逐条返回 JSON 事件：
    - `{"event":"delta","answer":"...","full_text":"..."}`：增量正文；
    - `{"event":"done","topic_closed":true|false}`：本轮完成；
    - `{"event":"error","message":"..."}`：业务或流式错误；
  - 前端以 `full_text` 驱动气泡实时展示；收到 `done` 后结束本轮；
  - 若连接失败、异常断开、超时、返回 `error`，统一回退 `POST /messages`（blocking）。

### WebSocket 鉴权与边界

- WebSocket 握手不依赖 FastAPI `Depends(settings.oauth2_scheme)`；由服务端在 WS 路由中读取 query `token` 并复用 JWT 校验逻辑解析用户。
- 鉴权失败、会话不存在、会话不可发送（智能体下架/删除、话题已结束）时，服务端发送 `event=error` 后主动关闭连接，不调用 Dify。
- 与 SSE 路径保持同等持久化语义：user 消息先落库；流正常结束后落库 assistant、更新 `dify_conversation_id`/`last_message_preview`/`is_topic_closed`，再发送 `done`。

## 约束与边界

- 管理端未上架的智能体在小程序侧不可见、不可建会话。
- 流式失败（网络中断、非 2xx）时以接口与落库一致性为准：流未正常结束则不写入 assistant；用户消息在流开始时已写入。
