# 功能设计

## 多标签展示（系统创建 / 微信用户）

- 表 `vadmin_auth_user` 增加 **`is_system_created`**（布尔，默认 `true`）：由后台「新增用户」、批量导入等路径创建时为 `true`；**仅由小程序微信授权首次自动建号**时为 `false`。微信合并已有账号时**不修改**该字段，以保留「先系统后微信」的双标签语义。
- 展示标签由后端计算 **`user_tags`** 字符串数组返回列表/详情：
  - 含 **「系统创建」**：`is_system_created == true`
  - 含 **「微信用户」**：`user_type == wechat` 或存在 `wx_nickname` / `wx_avatar` 等微信档案字段
- 与 `user_type` 单字段并存：`user_type` 仍表示当前主来源；`user_tags` 仅用于后台运营识别，**不替代**角色权限。

## 拉黑

- 字段 **`is_blocked`**（布尔，默认 `false`）。
- **登录拦截**：密码/短信登录校验、微信登录、`/token/refresh` 刷新 token、以及任意依赖 `validate_user` 的接口均在 `is_active` 之前检查拉黑；文案统一为「拉黑」相关提示。
- **管理接口**：`PUT /vadmin/auth/users/{id}/blocked`，请求体 `{ "is_blocked": true|false }`，权限与「更新用户」一致（`auth.user.update`）。禁止对当前登录用户及 id=1 超级管理员操作。

## 删除

- 沿用现有 **软删除**（`is_delete` + `is_active=false` 等），批量删除逻辑不变；删除前二次确认由管理端表格组件承担。

## API 与序列化

- 用户列表/详情使用 **`UserOut`**，在序列化时注入 **`user_tags`**。
- 列表查询增加可选参数 **`is_blocked`** 筛选。

# 数据模型

- `vadmin_auth_user.is_system_created`：`Boolean`，非空，默认 `true`。
- `vadmin_auth_user.is_blocked`：`Boolean`，非空，默认 `false`。
- 迁移：`hipp-or/hipp-api/alembic/versions_dev/a1b2c3d4e5f6_user_system_created_blocked.py`（`down_revision`: `e1f3a5b7c9d1`）。

# 约束与边界

- 不使用数据库外键；用户关联业务表（如 `mp_chat` 会话）的级联清理不在本迭代强制实现，以软删除用户为主。
- 存量「先系统后微信合并」账号若迁移前无 `is_system_created`，以迁移后字段为准；纯微信存量账号默认 `is_system_created=true` 时可能短暂出现双标签，可运营侧按需手工纠偏。
