# 背景

1. C 端用户需要在微信小程序中与智能助手对话；企业侧已通过 Dify 搭建智能体，并在本平台的「智能客服管理」中完成接入与上下架（参见 `hipp-or/doc/01-or/【智能客服管理】Dify智能体接入管理.md`）。本需求界定**小程序侧**的对话入口、列表与会话体验，以及**后端**与 Dify 标准 API 的对接范围。
2. 小程序工程位于 `hipp-or/hipp-uni`，对话收件箱在 `pages/chat/index`，单会话聊天页在 `subpkg/chat/chat`（分包以控制主包体积）。当前已有页面与请求占位，缺少与平台智能体、会话列表及 Dify 流式/完整回复等业务闭环一致的定义。
3. 后端采用 FastAPI，对话业务能力在 `hipp-or/hipp-api` 中实现（例如 `apps/mp_chat` 等约定目录）。用户体系与权限已具备；智能体接入与上架数据已可由管理端维护，小程序仅需按规则消费「已上架」智能体并持久化用户会话。
4. 「需求」「商业」两个场景 Tab 的入口职责、智能客服类型映射、先请求后端再会话等要求见独立原始需求 `hipp-or/doc/01-or/【小程序对话】需求与商业场景页.md`。
5. 与「话题结束、归档、同智能体多会话」相关的补充约定见 `hipp-or/doc/01-or/【小程序对话】用户智能对话——话题归档功能.md`；Dify 对接方式遵循官方「使用 API 开发」说明：[Developing with APIs](https://docs.dify.ai/zh/use-dify/publish/developing-with-apis）。

# 目标

1. 将小程序主导航/首屏定位为「对话」能力入口，布局与常见智能对话应用一致，便于用户理解与使用。
2. 在「对话」Tab 的列表中**仅展示用户已归档话题**（与话题归档功能一致），支持头像、名称与最近消息预览；**进行中对话与智能体入口**由「需求」「商业」等场景页承担，不得与归档列表混排（详见「要求」）。场景页与可服务智能体、类型与随机策略见 `hipp-or/doc/01-or/【小程序对话】需求与商业场景页.md`。
3. 对会话提供整理类操作（置顶、重命名、分享、删除等，在约束条件下可用）；**置顶**会话（含已归档话题）可上浮至列表前部，与列表排序规则一并满足。
4. 提供完整对话页：展示当前智能体或会话名称、返回列表并保持导航状态、消息输入与发送。
5. 由后端封装对 Dify 的调用与平台侧会话/智能体数据，保证与权限及上架策略一致。

# 要求

## 功能性要求

### 整体布局与入口

- 主界面标题为「对话」；导航栏标题同步为「对话」；右上角提供「搜索」入口（具体搜索范围与接口在开发设计中细化）。
- 内容区行为：展示**当前用户已归档话题**列表（`is_topic_closed=true` 的会话）；不提供已上架智能体固定入口行。用户发起或继续进行中对话须通过「需求」「商业」等场景 Tab（见 `小程序整体设计` 与开发设计）；场景页细则见 `hipp-or/doc/01-or/【小程序对话】需求与商业场景页.md`。

### 归档话题列表（对话 Tab）

- **范围（强制）**：列表**仅包含已归档话题**会话；不包含 `kind=agent` 智能体入口行，不包含未归档（进行中）会话行。
- **排序（强制）**：**置顶**的归档会话整体排在**非置顶**归档会话之前；同一置顶分区内按最近活跃时间倒序（新在前）；非置顶分区内同样按最近活跃倒序。具体字段与实现见 `hipp-or/doc/02-dr/04 小程序对话/小程序用户对话设计.md`。
- **数据来源**：当前用户下 `is_topic_closed=true` 且未软删除的会话，左连接智能体信息用于头像、名称与 `agent_status` 展示。
- **列表项展示**：头像、名称（`display_title` 等）、最近一条消息摘要；已下架/已删除智能体的历史会话仍展示，并标记状态。
  - **头像规则（强制）**：会话列表图标必须展示「对话对方头像」。用户与智能体会话时取当前智能体头像（不可用时回退会话快照头像）；用户与人工客服会话时取客服头像（不可用时使用人工客服默认头像占位）。禁止展示与对话对象无关的首字母/主题图标替代头像。
- **搜索 `q`**：仅匹配归档会话维度（标题、预览、快照名、关联智能体名称等），命中结果均为会话行。
- **长按**：对列表中的归档会话支持操作菜单：置顶/取消置顶、编辑对话名称、删除；已归档会话另含「继续对话」（与开发设计一致）。**置顶/取消置顶**文案规则与原先会话行一致（非置顶显示「置顶」，已置顶显示「取消置顶」）。

### 对话页

- 顶部展示当前**智能体名称**或当前**会话/对话记录名称**（以用户可理解的文案为准）。
- 左上角返回：返回「对话」列表，并**保持列表滚动位置等状态**（具体状态范围在开发设计中约定）。
- 底部为消息输入与发送区域，行为与通用智能对话一致（含加载中、失败提示等，见非功能性要求）。
- **底部输入区**：多行输入、随内容增高（有上限并可内部滚动）；唤起软键盘时输入区随页面上移，避免被键盘遮挡；发送以**图标按钮**为主（参考豆包等主流对话应用）；适配底部安全区（如 iPhone 底部横条），避免输入区与系统手势区重叠。
- **发送触发规则**：普通回车仅用于换行，**不得**将回车作为发送触发；发送仅允许通过发送按钮，或在支持外接键盘的场景下使用组合键（如 `Ctrl/Cmd + Enter`）。
- **用户消息展示与发送状态**：
  - 用户点击发送后，**须立即在会话消息区展示本条用户消息**（可先乐观展示再与后端结果对齐），不得出现「已发送但列表中无气泡/无正文」的情况。
  - 允许在气泡旁或角标展示发送状态：**发送中**、**成功**、**失败**；**失败**时用户可点击重试（重试语义见 `hipp-or/doc/02-dr/04 小程序对话/小程序用户对话设计.md`）。
- 若当前会话对应智能体为「已下架」或「已删除」，对话页仍可浏览历史消息，但输入区与发送动作必须禁用，并给出不可继续对话的明确提示文案。
- **助手消息展示宽度**：智能体回复通常篇幅较长，消息区横向应**尽量占满屏幕可视宽度**（与用户消息右对齐、限宽区分），避免右侧出现大块空白影响阅读。
- **与人工客服会话衔接**：当会话为「用户 ↔ 人工客服」类型（由需求类归档发起或平台约定的人工会话）时，对话页须支持用户**查看完整会话记录**与**继续发送消息**，交互与本文档「对话页」通用要求一致（输入区、发送状态、失败重试等）。人工客服侧在工作台对话页展示会话、查看记录与回复能力见 `hipp-or/doc/01-or/【人工客服】 需求类话题响应.md`。

### 后端与 Dify

- 在 `hipp-api` 中提供小程序所需的对话与会话相关接口，目录与模块划分以开发设计为准（如 `apps/mp_chat`）。
- 通过平台已存储的智能体配置（API 地址、密钥等）调用 Dify 开放接口；不得绕过平台已管理的配置直连未登记智能体。
- 会话与消息的持久化、与 `user_id` 等的关联方式由开发设计定义，须满足权限与数据隔离要求。
- 后端对发送接口必须执行智能体可用性校验：仅在智能体「已上架且未删除」时允许新对话或继续发送；对历史会话查询不做该限制。
- **`GET /mp/chat/inbox?kind=session`（对话 Tab 收件箱）**仅返回已归档话题会话（`kind=session`），排序落实上文「归档话题列表」规则；**不得**返回 `kind=agent` 行。不带 `kind` 的合并列表行为由开发设计说明（调试或其它客户端）。
- **结束状态解析兼容性（强制）**：后端判定/过滤 Dify 首行「结束状态」时，必须兼容 `结束状态false`、`结束状态 false`、`结束状态: false`、`结束状态：False` 等空格与中英文冒号变体，并对 `True/False` 做大小写不敏感处理（如 `true`、`TRUE`、`False`）。

## 非功能性要求

- 交互与视觉达到 C 端产品可用水准，符合**微信小程序**设计规范与审核相关实践。
- 与常见智能对话/IM 体验保持一致（如气泡区分、发送中/失败重试、长文本与 Markdown 等展示策略在开发设计中选型）。

## 问题

1. **本地发送后用户消息不展示**：根因为 `rich-text` nodes 使用非法标签名；已改为 `<text>` 并补充状态与重试（同上文档与 `hipp-uni/subpkg/chat/chat.vue`）。
2. **助手回复 Markdown 源码直出**：根因为对话页将助手消息按纯文本渲染，未执行 Markdown 格式化；已补充安全渲染策略并改为 Markdown 展示（见 `hipp-or/doc/91-qa/【小程序对话】助手回复Markdown源码直出.md`、`hipp-or/doc/02-dr/04 小程序对话/小程序用户对话设计.md`、`hipp-uni/subpkg/chat/chat.vue`）。
3. **后台更新智能体名称/头像后小程序对话展示不更新**：根因为列表与对话导航优先使用会话 `title`（创建时写入，未随智能体资料变更）；已增加接口 `display_title` 与 `onShow` 刷新会话元数据（见 `hipp-or/doc/91-qa/【小程序对话】后台更新智能体后对话页信息不刷新.md`）。
4. **收件箱 `kind` 返回与产品语义不一致**：进行中会话被返回为 `kind=session`，与「智能体固定入口，归档话题才是会话行」不一致；已调整为进行中统一 `kind=agent`（附带 `row.session`），仅归档返回 `kind=session`（见 `hipp-or/doc/91-qa/【小程序对话】收件箱kind返回与话题归档语义不一致.md`）。
5. **从对话页返回收件箱列表滚动位置丢失**：见 `hipp-or/doc/91-qa/【小程序对话】返回收件箱列表滚动位置丢失.md`、`hipp-uni/pages/chat/index.vue`。
6. **结束状态标记解析不稳**：Dify 可能返回 `结束状态false`（无空格）或 `true/false` 小写，导致状态判定与正文过滤不一致；已统一为大小写不敏感并兼容空格/冒号变体（见 `hipp-or/hipp-api/apps/mp_chat/dify_client.py`）。
