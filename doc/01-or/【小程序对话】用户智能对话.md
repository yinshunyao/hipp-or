# 背景

1. C 端用户需要在微信小程序中与智能助手对话；企业侧已通过 Dify 搭建智能体，并在本平台的「智能客服管理」中完成接入与上下架（参见 `hipp-or/doc/01-or/【智能客服管理】Dify智能体接入管理.md`）。本需求界定**小程序侧**的对话入口、列表与会话体验，以及**后端**与 Dify 标准 API 的对接范围。
2. 小程序工程位于 `hipp-or/hipp-uni`，对话相关页面在 `hipp-or/hipp-uni/pages/chat`。当前已有页面与请求占位，缺少与平台智能体、会话列表及 Dify 流式/完整回复等业务闭环一致的定义。
3. 后端采用 FastAPI，对话业务能力在 `hipp-or/hipp-api` 中实现（例如 `apps/chat` 等约定目录）。用户体系与权限已具备；智能体接入与上架数据已可由管理端维护，小程序仅需按规则消费「已上架」智能体并持久化用户会话。
4. Dify 侧已就绪，定制化程序以**接入**为主。对接方式遵循 Dify 官方「使用 API 开发」说明：[Developing with APIs](https://docs.dify.ai/zh/use-dify/publish/developing-with-apis)。

# 目标

1. 将小程序主导航/首屏定位为「对话」能力入口，布局与常见智能对话应用一致，便于用户理解与使用。
2. 在列表中展示**已上架**智能体及用户历史对话，支持头像、名称与最近消息预览，并对会话提供整理类操作（置顶、重命名、分享、删除等，在约束条件下可用）。
3. 提供完整对话页：展示当前智能体或会话名称、返回列表并保持导航状态、消息输入与发送。
4. 由后端封装对 Dify 的调用与平台侧会话/智能体数据，保证与权限及上架策略一致。

# 要求

## 功能性要求

### 整体布局与入口

- 主界面标题为「对话」；导航栏标题同步为「对话」；右上角提供「搜索」入口（具体搜索范围与接口在开发设计中细化）。
- 内容区行为：
  - 若存在**正在进行中的会话**（定义见开发设计，如当前线程未结束且用户未主动退出），进入对话页并**恢复/保持**会话状态；
  - 若无进行中的会话，展示「智能助手」列表（交互形态类似好友列表）。

### 智能助手列表

- 数据来源：**已上架**的全部智能体（与管理平台上架状态一致），以及当前用户的**对话记录**。
- 列表项展示：头像、名称、最近一条消息摘要（或时间等，以实现设计为准）。
- 历史会话保留可见性：若某会话对应智能体已下架或已删除，客户端仍需展示该会话与历史消息，并在列表/会话页明确标记为「已下架」或「已删除（历史）」。
- **长按**：
  - **对话记录**：在条件满足时（如仅针对用户自有会话、非系统限制场景等，开发设计明确）支持弹出操作菜单，包含：置顶/取消置顶、编辑对话名称、分享对话、删除。
  - **智能体**条目：不弹出上述会话操作框（点击进入对话即可，或按开发设计补充极简说明）。

### 对话页

- 顶部展示当前**智能体名称**或当前**会话/对话记录名称**（以用户可理解的文案为准）。
- 左上角返回：返回「对话」列表，并**保持列表滚动位置等状态**（具体状态范围在开发设计中约定）。
- 底部为消息输入与发送区域，行为与通用智能对话一致（含加载中、失败提示等，见非功能性要求）。
- **底部输入区**：多行输入、随内容增高（有上限并可内部滚动）；唤起软键盘时输入区随页面上移，避免被键盘遮挡；发送以**图标按钮**为主（参考豆包等主流对话应用）；适配底部安全区（如 iPhone 底部横条），避免输入区与系统手势区重叠。
- **用户消息展示与发送状态**：
  - 用户点击发送后，**须立即在会话消息区展示本条用户消息**（可先乐观展示再与后端结果对齐），不得出现「已发送但列表中无气泡/无正文」的情况。
  - 允许在气泡旁或角标展示发送状态：**发送中**、**成功**、**失败**；**失败**时用户可点击重试（重试语义见 `doc/02-dr/04 小程序对话/小程序用户对话设计.md`）。
- 若当前会话对应智能体为「已下架」或「已删除」，对话页仍可浏览历史消息，但输入区与发送动作必须禁用，并给出不可继续对话的明确提示文案。
- **助手消息展示宽度**：智能体回复通常篇幅较长，消息区横向应**尽量占满屏幕可视宽度**（与用户消息右对齐、限宽区分），避免右侧出现大块空白影响阅读。

### 后端与 Dify

- 在 `hipp-api` 中提供小程序所需的对话与会话相关接口，目录与模块划分以开发设计为准（如 `apps/chat`）。
- 通过平台已存储的智能体配置（API 地址、密钥等）调用 Dify 开放接口；不得绕过平台已管理的配置直连未登记智能体。
- 会话与消息的持久化、与 `user_id` 等的关联方式由开发设计定义，须满足权限与数据隔离要求。
- 后端对发送接口必须执行智能体可用性校验：仅在智能体「已上架且未删除」时允许新对话或继续发送；对历史会话查询不做该限制。

## 非功能性要求

- 交互与视觉达到 C 端产品可用水准，符合**微信小程序**设计规范与审核相关实践。
- 与常见智能对话/IM 体验保持一致（如气泡区分、发送中/失败重试、长文本与 Markdown 等展示策略在开发设计中选型）。

# 已完成工作项

## 需求

1. 小程序用户智能对话：收件箱与对话页、后端 `/mp/chat` 与 Dify blocking 对接、数据表 `vadmin_chat_session` / `vadmin_chat_message`（详见 `doc/02-dr/04 小程序对话/小程序用户对话设计.md`）。
2. **对话页用户消息展示与发送状态**：用户消息用 `<text>` 展示（避免微信 `rich-text` 非法节点导致正文不渲染）；乐观追加消息；发送中/已发送/失败与失败重试（见设计 `doc/02-dr/04 小程序对话/小程序用户对话设计.md`、测试 `doc/03-tr/04 小程序对话/小程序用户对话测试设计.md`）。
3. **对话页助手消息横向铺满**：助手回复气泡占满屏幕可视宽度（与用户右对齐限宽区分），长文阅读时右侧不留大块空白（见同上设计/测试与 `hipp-uni/pages/chat/chat.vue`）。

## 问题

1. **本地发送后用户消息不展示**：根因为 `rich-text` nodes 使用非法标签名；已改为 `<text>` 并补充状态与重试（同上文档与 `hipp-uni/pages/chat/chat.vue`）。
2. **助手回复 Markdown 源码直出**：根因为对话页将助手消息按纯文本渲染，未执行 Markdown 格式化；已补充安全渲染策略并改为 Markdown 展示（见 `doc/91-qa/【小程序对话】助手回复Markdown源码直出.md`、`doc/02-dr/04 小程序对话/小程序用户对话设计.md`、`hipp-uni/pages/chat/chat.vue`）。
