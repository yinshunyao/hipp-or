# 背景

1. 微信小程序在主导航中提供「需求」「商业」两个场景 Tab，作为用户与对应类型智能体进行**进行中对话**的入口；与「对话」Tab 仅展示**已归档话题**列表的职责分离，整体关系见 `hipp-or/doc/01-or/【小程序对话】用户智能对话.md`。
2. 企业侧已通过 Dify 搭建智能体，并在本平台「智能客服管理」中完成接入与上下架（参见 `hipp-or/doc/01-or/【智能客服管理】Dify智能体接入管理.md`）。「需求」「商业」所绑定的智能体**不以 Dify 应用名称为唯一依据**，而应以管理端配置的**智能客服类型**（「需求分析」「商业评估」）为准；由**后端**在可服务集合内按类型解析并（同类型多智能体时）**随机选定 1 个**，小程序在进入场景时先请求后端再创建或续聊会话。
3. Dify 对接方式遵循官方「使用 API 开发」说明：[Developing with APIs](https://docs.dify.ai/zh/use-dify/publish/developing-with-apis）。

# 目标

1. 用户从「需求」或「商业」进入时，**必须先**经后端确认当前可服务的智能体，再创建会话或绑定已有进行中会话，**禁止**仅依赖客户端硬编码名称匹配作为唯一路由方式。
2. **类型对应（强制）**：**需求**场景对应**需求分析**；**商业**场景对应**商业评估**（类型取值与 `【智能客服管理】Dify智能体接入管理.md` 一致）。
3. 同一类型下**有多个**已上架候选时，由**服务端在同类型内随机选定 1 个** `agent_id` 返回；该类型下**无可服务**智能体时，场景页须明确提示不可用，不得静默失败。
4. 若用户在该场景下已有**进行中**会话且仍可续聊，应优先续聊；原绑定智能体已下架/删除而同类型仍有其他上架智能体时的策略由开发设计约定。

# 要求

## 功能性要求

### 场景页「需求」「商业」与可服务智能体（类型与随机）

- **先请求后端**：进入「需求」或「商业」场景页、准备发起或继续进行中对话时，**必须先**调用后端提供的接口，获取当前**可服务**的智能体信息（仅 `status=published` 且未软删除），再据此创建会话或绑定已有进行中会话。**禁止**仅依赖 Dify 应用名称在客户端硬编码匹配（如固定某几个名称）作为唯一路由方式。
- **类型对应（强制）**：管理端「智能客服类型」与场景一一对应——**需求**场景对应类型 **需求分析**；**商业**场景对应类型 **商业评估**（类型取值与 `hipp-or/doc/01-or/【智能客服管理】Dify智能体接入管理.md` 一致）。
- **每类型 1 个可服务实例**：后端在**同一类型**下筛选已上架智能体；若该类型下**有多个**候选，由**服务端在同类型内随机选定 1 个** `agent_id` 返回给小程序，作为本场景当前使用的智能体。若该类型下**无**可服务智能体，场景页应明确提示不可用，不得静默失败；具体接口路径、响应字段与错误码在开发设计中约定。
- **会话延续**：若用户在该场景下已有**进行中**会话（`is_topic_closed=false`）且会话关联的智能体仍属于同一类型且仍可服务，应优先续聊该会话；若原绑定智能体已下架/删除而同类型仍有其他上架智能体，由后端策略与开发设计约定（例如是否允许切换到新随机智能体新开会话）。

### 话题完成后的继续对话（开启新话题）

- **仅提示不禁用输入**：当聊天过程中服务端判定话题完成（会话 `is_topic_closed=true` 或流式结束事件 `mp_topic.topic_closed=true`），页面底部仅展示「本话题已结束」类消息提示，**用户仍可以继续输入并发送**以开启新的话题；禁止出现“无法继续发送/直接禁用输入框”导致用户无法开启新话题的交互。
- **发送即新建话题**：在话题已结束且智能体仍可用（`agent_status=active`）时，用户点击发送应由客户端（或服务端约定策略）**自动创建新会话**并将该条消息发送到新会话，作为新话题的首条消息；原已结束话题保持归档不变。

### 助手消息的 Markdown 展示（需求/商业场景页）

- **原生渲染优先**：智能问答（助手）回复支持 Markdown（标题、列表、引用、行内/块代码、链接、表格），前端应将 Markdown 转换为小程序 `rich-text` 支持的 **HTML 标签集合**进行展示，尽量使用小程序原生渲染能力完成（避免依赖 WebView）。
- **安全与兼容**：渲染前需进行基础安全处理（先转义原始 HTML，再生成受控标签）；链接仅允许 `http/https`；表格每行必须生成完整 `<tr><td>…</td></tr>` 以避免小程序渲染错位。

### 与「对话」Tab 及归档列表的边界

- 「对话」Tab 的收件箱**仅**展示已归档话题会话；**进行中**对话由本需求所述场景页（及同类扩展）承担，**不得**与归档列表混排。详见 `hipp-or/doc/01-or/【小程序对话】用户智能对话.md`。
- 场景页内已归档话题的展示顺序、分页与进入会话详情等，在 `hipp-or/doc/02-dr/04 小程序对话/小程序用户对话设计.md` 中约定。

## 非功能性要求

- 场景页在智能体不可用时须有明确空态/错误提示，符合微信小程序设计规范与审核相关实践。
