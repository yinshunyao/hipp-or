# 背景

1. 用户与同一智能体可能进行多轮独立「话题」沟通；若仅依赖单一会话线程，历史与进行中的对话混杂，不利于在收件箱中区分、回顾与继续某一话题。
业务上需要在智能体判定某一话题已结束时，将该段对话**归档**为一条可识别的会话记录，并允许用户在同一智能体下开启新的对话线程。
2. 平台已在后端约定：Dify 助手回复正文首行可携带「结束状态 True / False」以表达需求沟通是否完成，并在落库与展示前滤除该行，
避免混入用户可见正文（见 `hipp-or/doc/01-or/【小程序对话】用户智能对话——后端处理.md`）。
**话题归档**是在此能力之上，当识别为「结束状态 True」时，在小程序侧完成会话形态切换、列表展示与后续新会话流程，并与收件箱、对话页既有能力（如长按会话操作、顶部标题展示）衔接。
3. 收件箱与对话页的整体行为以 `hipp-or/doc/01-or/【小程序对话】用户智能对话.md` 为准；
本文件仅补充与**话题结束、归档、会话分裂与标题**相关的范围，不重复定义通用列表与发送能力。

# 目标

1. 在助手回复被判定为「话题结束」（结束状态 True）时，向用户提供明确的**话题已结束**反馈，且在**最后一条流式内容展示完毕之后**再完成归档相关交互，避免打断阅读。
2. 将已结束话题固化为**独立的会话记录**：在收件箱中可与其他话题区分展示（默认在对应智能体条目下形成可区分的会话行或等价结构，具体布局以实现设计为准），并支持用户从历史入口再次进入该话题内容。
3. 话题归档后，用户与同一智能体的**新的用户消息**按**新会话**处理（清空当前对话区的「进行中」上下文，与 Dify 多轮 `conversation_id` 的衔接策略在开发设计中定义），避免新需求与已归档话题混在同一可视线程中。
4. 保证话题维度下的**标题**符合用户预期：初始标题来自平台与 Dify 的约定产出（如首轮或首次有效命名来源），并支持用户在对话页与列表中按既有规则修改（与 `【小程序对话】用户智能对话.md` 中会话重命名、长按操作一致）。
5. 支持在条件满足时**对已有归档话题追加对话**（延续原话题），交互入口与约束与主需求文档中的长按等操作保持一致，且不破坏「话题结束」与「新会话」的语义边界。

# 要求

## 功能性要求

### 结束判定与交互时序

- 当服务端在流式或非流式路径中解析到助手内容首行为「结束状态 True」（及与后端处理文档一致的格式变体）时，应视为**本话题可归档**的判定条件之一；具体是否仅依赖该标识、是否与业务字段组合判定，以实现设计与 Dify 编排为准。
- **须等待**当前轮次助手回复的流式输出**全部结束**（最后一条消息展示完毕）后，再触发面向用户的「话题结束 / 归档」相关提示或状态切换，不得在流未完成时提前打断或覆盖消息区。
- 服务端须在适当时机将「本话题已结束」的可消费信息传递给小程序（例如随流结束后的响应体字段、SSE 内约定事件或等价机制，**不**要求在本 OR 中固定技术形态），使客户端能驱动 UI；客户端须展示可理解的**话题已结束**类提示文案。
- **微信小程序**：当会话为只读且需在底部展示说明（含话题已结束、智能体下架/删除等）时，优先采用 `@vant/weapp` 的 `NoticeBar`：`wrapable`、**禁止**横向滚动（`scrollable` 为 false），以多行公告条形式展示文案；其它端可保持等价信息的通用样式。
- 话题已结束且展示「联系人工客服」时，**提示文案与按钮不得在同一平面重叠遮挡**：按钮应位于提示条**下方单独一行**（或等价布局），保证提示全文可读；**底部整段（提示 + 按钮）高度随内容自适应**，除输入栏外不重复叠加底部安全区内边距，避免出现大块无效留白。

### 归档后会话形态与列表

- 归档后，对话页顶部标题区应从展示**智能体名称**转为展示**该话题/会话的名称**（与 `vadmin_chat_session.title` 或产品文案一致），使用户明确当前查看的是某一历史话题而非进行中的「开聊」态。
- 用户再次打开小程序进入收件箱时，已归档话题须作为**独立会话记录**呈现，并与「尚未归档的进行中会话」「仅展示智能体入口」等状态可区分；默认展示关系为在**对应智能体下方或分组内**列出（与 `【小程序对话】用户智能对话.md` 中列表结构一致，细节由开发设计细化）。
- 同一智能体下，完成归档后**新开对话**时，对话区应清空或切换为**新会话**展示，新消息走新会话流程；已归档话题仅通过历史会话入口查看，不得与新区消息串混在同一可视线程中（除非用户明确选择「追加到该话题」）。
- 已归档话题在对话详情页打开时，底部输入框与发送按钮**默认不展示**（而非仅禁用）；仅在用户执行「继续对话/重新打开话题」后恢复输入能力。

### 标题与命名

- **初始会话/话题标题**须来自与 Dify 及平台约定的来源（例如首轮对话生成的标题、或接口返回的命名字段；具体字段与截取规则在开发设计中写明），不得仅依赖客户端随意占位字符串作为最终业务标题。
- 对话页顶部须支持用户**修改话题名称**（点击标题进入编辑或等价交互），并与列表中「编辑对话名称」能力一致或互补，避免两处修改结果冲突。

### 追加与延续

- 用户对某一已存在会话（含已归档话题）的**追加对话**、长按菜单等，遵循 `hipp-or/doc/01-or/【小程序对话】用户智能对话.md` 的约束；本需求不新增与之冲突的操作定义。

## 非功能性要求

- 话题结束与归档相关提示、标题切换须符合微信小程序交互与审核习惯，文案清晰、可访问。
- 与流式推送、会话持久化、错误恢复的配合须避免产生「未归档却显示已结束」「已归档仍可向旧 Dify 线程误发」等不一致状态；异常与重试边界由开发设计与测试设计覆盖。
