# 背景

1. 小程序登录方式需与运营与安全策略对齐：**不再使用微信授权登录**，改为用户主动输入手机号并完成短信验证，流程可预期、可审计。
2. 降低新用户门槛：首次使用小程序的用户在通过短信验证后应能**直接获得账号**，无需管理员预开户或走微信手机号授权。
3. 登录成功后是用户进入「话题 / 需求 / 商业 / 我的」核心页面的前置条件，登录过程需要稳定、可追踪、可回退。

# 目标

1. 小程序登录**默认**为「手机号 + 短信验证码」：用户获取验证码、输入验证码后完成登录。
2. 短信验证码校验通过时，若该手机号在系统中**不存在**，则**自动创建新用户账号**并完成登录（与老用户共用同一套登录态与权限模型约束）。
3. 保留「账号密码登录」作为**可选补充入口**（与现有系统用户习惯兼容），与短信登录可在登录页切换。
4. 小程序内**移除微信登录**相关入口与流程（不再引导用户使用微信授权登录；是否保留服务端接口仅作兼容由开发设计约定）。

# 要求

## 功能性要求

### 登录入口与方式

- 登录页**默认展示**短信登录：手机号输入、验证码输入、「获取验证码」操作、登录提交。
- 登录页须允许用户切换到「账号密码登录」（手机号或用户名 + 密码，密码可显隐切换），切换后输入状态行为在设计文档中明确（避免误丢已填手机号等）。
- 登录页**不得**再提供「微信授权登录」按钮或与微信一键登录等价的入口。

### 短信验证码登录

- 手机号须符合平台校验规则；验证码发送、有效期、重发间隔、单日次数上限等由开发设计结合短信服务商能力约定，并在异常时给出明确提示（如发送过频、验证码错误、已过期）。
- **校验通过后**：
  - 若该手机号**已存在**用户：按现有账号规则登录（状态、拉黑、冻结、权限等校验与设计一致）。
  - 若该手机号**不存在**：**创建新用户**并登录；新用户的默认字段（如 `user_type`、昵称、初始密码策略、`is_staff` 等）在开发设计中与后台用户体系对齐，须保证创建后即可通过本次登录态访问小程序已授权业务能力。
- 短信登录仅支持**中国大陆手机号**（或与现有 `telephone` 字段约束一致），非法格式拒绝发送。

### 账号密码登录

- 用户输入账号与密码后点击登录，前端调用后端登录接口进行校验（与历史「手机号或用户名 + 密码」语义一致）。
- 登录失败时，页面需明确提示失败原因（如账号不存在、密码错误、账号状态异常），不得静默失败。
- 连续失败场景的限制策略（如频控、锁定时长）沿用现有实现或在设计文档中补充。

### 登录成功与会话建立

- 不论采用哪种登录方式，只有在后端返回成功并下发可用登录态后，才可进入小程序主流程。
- 登录成功后应跳转到业务首页（具体落地页以后续设计约定），并在后续请求中带上登录态。
- 登录态失效时，用户再次进入受限页面应被引导重新登录。

## 非功能性要求

- 登录页交互符合微信小程序 UI 规范；短信发送与倒计时、错误提示具备清晰反馈。
- 采集与处理遵循最小必要原则：短信登录以手机号 + 验证码为必要字段。
- 登录关键链路可观测：至少保留登录方式、结果、失败原因类别等基础日志字段，便于排障；自动建号须有可追溯记录（与设计一致）。
