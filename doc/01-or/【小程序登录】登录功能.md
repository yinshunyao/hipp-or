# 背景

1. 当前小程序尚未形成统一、可落地的登录需求定义，前后端对登录入口、授权项、账号创建与绑定规则缺少一致约束，导致实现容易反复。
2. 小程序业务需要同时支持传统账号登录与微信授权登录，且需满足微信小程序常见交互习惯与用户对隐私授权可感知、可选择的预期。
3. 登录成功后是用户进入「话题 / 需求 / 商业 / 我的」核心页面的前置条件，登录过程需要稳定、可追踪、可回退。

# 目标
1. 支持「手机号/用户名 + 密码」登录，提供密码可见切换，降低输入错误率并提升登录成功率。
2. 支持微信授权登录，按业务所需获取头像、昵称、手机号，并由后端完成账号创建/绑定。
3. 建立统一登录成功标准：后端处理成功并返回可用登录态后，用户可进入小程序首页与业务流程。

# 要求

## 功能性要求

### 登录入口与方式

- 登录页必须同时提供两种方式：`账号密码登录` 与 `微信授权登录`。
- 账号密码登录支持手机号或用户名作为账号输入值，密码输入默认掩码显示。
- 密码框需提供「显示/隐藏」切换图标；切换后仅影响当前输入框展示，不改变密码真实值。

### 账号密码登录

- 用户输入账号与密码后点击登录，前端调用后端登录接口进行校验。
- 登录失败时，页面需明确提示失败原因（如账号不存在、密码错误、账号状态异常），不得静默失败。
- 连续失败场景的限制策略（如频控、验证码、锁定时长）由后续设计文档明确，但 OR 要求必须预留该能力。

### 微信授权登录

- 登录页需提供微信授权登录按钮，触发后以底部弹出列表方式展示待申请信息，顺序为：头像、昵称、手机号。
- 用户确认授权后，前端展示已获取的授权信息摘要，并提交给后端。
- 后端必须支持微信用户登录处理：根据授权信息创建新账号或绑定已有账号，并记录腾讯侧用户全局标识（如 openid/unionid，具体字段以后端设计为准）。
- 若用户拒绝授权或授权信息不完整，前端需提供可理解的提示，并允许用户改用账号密码登录。

### 登录成功与会话建立

- 不论采用哪种登录方式，只有在后端返回成功并下发可用登录态后，才可进入小程序主流程。
- 登录成功后应跳转到业务首页（具体落地页以后续设计约定），并在后续请求中带上登录态。
- 登录态失效时，用户再次进入受限页面应被引导重新登录。

## 非功能性要求

- 登录页交互符合微信小程序 UI 规范，按钮、输入框、授权弹层具备清晰状态反馈。
- 授权信息与登录态处理需遵循最小必要原则，仅采集业务必需字段。
- 登录关键链路需可观测：至少保留登录方式、结果、失败原因类别等基础日志字段，便于排障。
